Following shows the stages or steps in managing a sandbox. Let's assume the name of the sandbox is `yoda` and the AWS_PROFILE (provider environment) is dev

<Steps>

1. Create

	```sh
	tc create -s yoda -e dev
	```

2. Update

	```sh
	tc update -s yoda -e dev
	```
	or update specific components for faster iteration

	```sh
	tc update -s yoda -e dev -c ENTITY
	tc update -s yoda -e dev -c ENTITY/COMPONENT

	e.g
	tc update -s yoda -e dev -c routes
	tc update -s yoda -e dev -c functions/vars

	```

3. List

	```sh
	tc list -s yoda -e dev
	```

4. Invoke

	```sh
	tc invoke -s yoda -e dev -p payload.json
	```

5. Freeze

	Freeze any updates to the sandbox. Useful for testing a shared sandbox typically QA or if there are continuous deploys.

	```sh
	tc freeze -s yoda -e dev
	```

6. Snapshot

	Snapshot the manifests of a sandbox. The manifest is a self-contained executable representaion of the sandbox

	```sh
	tc snapshot -s yoda -e dev --manifest --save s3://bucket/key.json
	```

7. Unfreeze

	Unfreeze any updates to the sandbox.

	```sh
	tc unfreeze -s yoda -e dev
	```

8. Delete

	```sh
	tc delete -s yoda -e dev
	```

	To prune stale resources

	```sh
	tc prune -s yoda -e dev [--filter expr]
	```

9. Create from manifest

	Let's say we create another sandbox in staging

	```sh
	tc create -s yoda -e dev --from s3://bucket/key.json
	```

10. Route

	Route core traffic from another sandbox. This is how canaries can be implemented using tc.

	```sh
	tc route -s yoda -e dev --from <from-sandbox> --root ENTITY/NAME
	```


</Steps>
